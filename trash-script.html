Deploy do Voicebox na GCP — Passo a passo

Baseado no repositório: amdremessias/lab26-voicebox (Docker/compose). Porta típica: 17493.

Resumo e opções recomendadas

  • Se vai rodar TTS com PyTorch/CUDA e quer desempenho: use Compute Engine com GPU (ou GKE com nodes GPU).
  • Se é só teste/CPU: VM comum (Ubuntu) é mais barata.
  • Mais simples: usar o docker-compose.yml do repo.
  • Persistência: reserve disco e use VOICEBOX_MODELS_DIR para modelos.

Recomendação prática: comece com Compute Engine + Docker. Depois, se precisar escalar, migre para GKE.

Opção A — Compute Engine (VM com GPU) — recomendado para inferência rápida

1) Pré-requisitos no projeto GCP

  • Projeto com faturamento ativo.
  • Cota de GPU aprovada na região/zone escolhida.
  • API do Compute Engine habilitada.

2) Criar a VM (hardware e disco)

  • Sistema: Ubuntu 22.04 ou 24.04.
  • Tipo de máquina: ex. n1-standard-8 (ajuste ao seu uso).
  • GPU: T4/A10/A100 conforme cota e custo.
  • Disco persistente: ex. 200 GB+ (modelos podem ocupar muito espaço).
  • Firewall: permitir TCP 17493 se você for acessar externamente.

3) Preparar a VM (Docker + suporte a GPU)

  • Instalar drivers NVIDIA.
  • Instalar Docker Engine e Docker Compose.
  • Instalar/configurar nvidia-container-toolkit para expor a GPU aos containers.

4) Obter o código e subir com Docker Compose

  • Clonar o repositório.
  • Rodar docker compose up -d no diretório do projeto.
  • Se o compose não estiver configurado para GPU, ajustar para usar runtime NVIDIA ou iniciar com suporte a GPU.
  • Montar volume/pasta persistente e setar VOICEBOX_MODELS_DIR.

5) Exposição, SSL e produção

  • Para acesso público com HTTPS: usar proxy reverso (Nginx) + certificado, ou um Load Balancer HTTPS do GCP.
  • Evite expor a API sem proteção: restrinja por IP/VPC, adicione autenticação, ou coloque atrás de gateway.
  • Garanta auto-restart: configure restart: unless-stopped no compose e/ou systemd.

Opção B — GKE com nodes GPU (produção / escalável)

  • Criar cluster e node pool com GPU.
  • Deploy via Deployment com requests/limits de GPU.
  • Persistência via PersistentVolumeClaim para modelos.
  • Exposição via Service/Ingress e Load Balancer.
  • Nota: Cloud Run normalmente não atende para GPU; para aceleração, prefira Compute Engine/GKE.

Opção C — Apenas CPU (teste)

  • Criar VM sem GPU, instalar Docker, rodar docker compose up -d.
  • Funciona, mas a inferência pode ficar significativamente mais lenta.

Checklist de riscos e pontos críticos

  • Disco: modelos grandes exigem espaço; use disco persistente e pasta dedicada.
  • Cota/região: GPU pode não estar disponível na zone escolhida.
  • Segurança: não exponha endpoints sem controle de acesso.
  • Compatibilidade CUDA: valide versões de CUDA/PyTorch do container/projeto.
