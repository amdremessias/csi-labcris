Tutorial: Configurando Nginx Proxy Manager com DuckDNS e SSL para o Serviço Ticketz

Este tutorial detalhado guiará você através do processo de configuração do Nginx Proxy Manager (NPM) com um subdomínio DuckDNS e certificados SSL via DNS Challenge. O objetivo é resolver problemas de acesso a recursos como microfone em aplicações web (como o Ticketz) que exigem uma conexão HTTPS segura.

Introdução

Ao desenvolver e hospedar aplicações web, especialmente aquelas que utilizam recursos sensíveis do navegador como o microfone (através da API enumerateDevices), é crucial garantir que a conexão seja segura (HTTPS). Sem um certificado SSL válido, os navegadores modernos bloqueiam o acesso a esses recursos, resultando em erros como TypeError: Cannot read properties of undefined (reading 'enumerateDevices').

Este guia aborda a integração do Nginx Proxy Manager, uma ferramenta poderosa para gerenciar proxies reversos e certificados SSL, com o serviço de DNS dinâmico DuckDNS. Utilizaremos o método DNS Challenge para a emissão de certificados Let's Encrypt, que permite a validação do domínio sem a necessidade de abrir portas como a 80 ou expor diretamente o IP público do seu servidor.

Pré-requisitos

Antes de iniciar, certifique-se de ter os seguintes itens configurados:

  • Servidor: Uma máquina virtual (VM) ou servidor com Docker e Docker Compose instalados (ex: Ubuntu em Proxmox).
  • Nginx Proxy Manager (NPM): Uma instância do NPM já em execução ou pronta para ser configurada via Docker Compose.
  • DuckDNS: Uma conta ativa no DuckDNS com um ou mais subdomínios configurados (ex: novochat.duckdns.org).
  • Serviço Web: O serviço que você deseja expor (neste caso, o Ticketz ou chat) rodando em containers Docker e acessível internamente.
  • NetBird (Opcional): Se você utiliza o NetBird para acesso à rede interna, certifique-se de que o IP do seu servidor esteja configurado corretamente.

Passo 1: Obtenha seu Token do DuckDNS

O token do DuckDNS é essencial para que o Nginx Proxy Manager possa se comunicar com o serviço DuckDNS e validar a propriedade do seu domínio para a emissão do certificado SSL.

  1. Acesse o site oficial do DuckDNS.
  2. Faça login com suas credenciais.
  3. Localize e copie o seu Token (uma sequência longa de letras e números). Guarde-o em segurança, pois ele será usado nas próximas etapas.

Exemplo de Token: 26887e1b-e75c-43d8-9d16-5f11135bb84d

Passo 2: Configurar o Docker Compose para Nginx Proxy Manager

Se você ainda não tem o Nginx Proxy Manager configurado, use o seguinte docker-compose.yml para iniciá-lo. Este arquivo garante a persistência dos dados e a exposição das portas necessárias.

  1. Crie um diretório para o NPM no seu servidor (ex: mkdir nginx-proxy).
  2. Dentro deste diretório, crie um arquivo chamado docker-compose.yml e adicione o seguinte conteúdo:

```yaml version: '3.8'

services: npm: image: 'jc21/nginx-proxy-manager:latest' container_name: nginx-proxy-manager restart: unless-stopped ports: - '80:80' # Porta HTTP padrão - '81:81' # Painel de Administração - '443:443' # Porta HTTPS padrão volumes: - ./data:/data - ./letsencrypt:/etc/letsencrypt environment: DISABLE_IPV6: 'true' # Opcional: Adicione servidores DNS para resolver problemas de comunicação externa # dns: # - 8.8.8.8 # - 1.1.1.1

networks: default: name: nginx-proxy # Se a rede 'nginx-proxy' foi criada manualmente (docker network create), use external: true # external: true ```

  1. Suba o container: No terminal, dentro do diretório nginx-proxy, execute:

    bash docker compose up -d

  2. Acesse o Painel do NPM: Abra seu navegador e acesse o IP do seu servidor (ou IP do NetBird) na porta 81 (ex: http://seu_ip:81).

    • Login padrão: admin@example.com
    • Senha padrão: changeme

    Altere as credenciais padrão imediatamente após o primeiro login.

Passo 3: Configurar o Docker Compose do Serviço Ticketz para integração com NPM

Para que o Nginx Proxy Manager possa rotear o tráfego para o seu serviço Ticketz, ambos precisam estar na mesma rede Docker. Além disso, o serviço Ticketz não deve expor suas portas diretamente para o host, mas sim para a rede interna do Docker.

  1. Crie a rede externa (se ainda não existir):

    bash docker network create nginx-proxy

  2. Edite o docker-compose.yaml do seu serviço Ticketz:

    Localize o arquivo docker-compose.yaml do seu serviço Ticketz (ex: /opt/ticketz-docker-local/docker-compose.yaml).

    • Adicione a rede nginx-proxy como externa: Dentro da seção networks, adicione a declaração para nginx-proxy:

      yaml networks: ticketz: nginx-proxy: external: true

    • Ajuste o serviço frontend: Remova a seção ports e adicione expose para a porta interna 80. Certifique-se de que o serviço frontend também esteja conectado à rede nginx-proxy.

      yaml frontend: image: ghcr.io/ticketz-oss/ticketz-frontend:latest volumes: - backend_public:/var/www/backend-public env_file: - .env-frontend # Retire as linhas de 'ports' e use 'expose' expose: - 80 restart: always depends_on: - backend networks: - ticketz - nginx-proxy # Adicione esta linha

    • Ajuste o serviço backend (se necessário): Se o NPM precisar se comunicar diretamente com o backend, adicione-o à rede nginx-proxy também.

      yaml backend: image: ghcr.io/ticketz-oss/ticketz-backend:latest volumes: - ./backend/config:/usr/src/app/config - backend_public:/usr/src/app/public - backend_private:/usr/src/app/private - ./migrations:/usr/src/app/migrations env_file: - .env-backend restart: always networks: - ticketz - nginx-proxy # Adicione esta linha depends_on: - postgres - redis

    • Verifique a indentação: O YAML é sensível à indentação. Use apenas espaços (nunca tabs) e mantenha o alinhamento correto. Erros de indentação podem causar falhas no docker compose.

  3. Aplique as mudanças: No terminal, no diretório do seu serviço Ticketz, execute:

    bash docker compose up -d --force-recreate

    Isso recriará os containers com as novas configurações de rede.

Passo 4: Configurar o Proxy Host no Nginx Proxy Manager

Agora que o NPM está rodando e o serviço Ticketz está na mesma rede, você precisa criar um Proxy Host no NPM para direcionar o tráfego externo para o seu frontend do Ticketz.

  1. Acesse o painel do NPM (porta 81).
  2. Vá em Hosts > Proxy Hosts.
  3. Clique em Add Proxy Host.
  4. Na aba Details, preencha da seguinte forma:
    • Domain Names: novochat.duckdns.org (ou o seu subdomínio DuckDNS).
    • Scheme: http
    • Forward Hostname / IP: frontend (Este é o nome do serviço frontend no seu docker-compose.yaml do Ticketz).
    • Forward Port: 80 (Esta é a porta interna do container frontend).
    • Websockets Support: Marque esta opção (essencial para aplicações de chat como o Ticketz).

Passo 5: Configurar o Certificado SSL via DNS Challenge (DuckDNS) no NPM

Esta etapa é crucial para obter o cadeado verde e habilitar o HTTPS, resolvendo o problema do microfone.

  1. Ainda na janela de configuração do Proxy Host, clique na aba SSL.
  2. Em SSL Certificate, selecione Request a new SSL Certificate.
  3. Marque a opção Use a DNS Challenge.
  4. Em DNS Provider, escolha DuckDNS na lista suspensa.

  5. No campo DNS Provider Configuration que aparecerá, insira o seu token do DuckDNS no formato:

    ini dns_duckdns_token = seu_token_aqui

    Exemplo: dns_duckdns_token = 26887e1b-e75c-43d8-9d16-5f11135bb84d

  6. Propagation Seconds: Pode deixar o padrão ou aumentar para 60 ou 120 segundos para dar mais tempo para o DuckDNS propagar as mudanças antes do Certbot verificar.

  7. Aceite os termos do Let's Encrypt e clique em Save.

Solução de Problemas (Troubleshooting) - Erros na Geração do Certificado

Se você encontrar erros como "KO" ou "Internal Error" durante a geração do certificado, verifique o seguinte:

  • Sintaxe do Token: Certifique-se de que o token está inserido exatamente como dns_duckdns_token = seu_token_aqui, sem espaços extras ou caracteres incorretos. O NPM é sensível a isso.
  • Subdomínio Existente: O erro KO do DuckDNS geralmente significa que o subdomínio (novochat.duckdns.org) não está criado ou não está vinculado à sua conta no painel do DuckDNS. Acesse o site do DuckDNS e crie o subdomínio se ele ainda não existir.
  • DNS do Docker: Se o erro persistir, o container do NPM pode estar com dificuldades para resolver nomes externos. Considere adicionar servidores DNS públicos (como Google 8.8.8.8 ou Cloudflare 1.1.1.1) ao seu docker-compose.yml do NPM, conforme o exemplo comentado no Passo 2.

Passo 6: Vincular o Certificado ao Proxy Host e Forçar SSL

Após a geração bem-sucedida do certificado, você precisa vinculá-lo ao seu Proxy Host e configurar o NPM para forçar o uso de HTTPS.

  1. No painel do NPM, vá em Hosts > Proxy Hosts.
  2. Edite o Proxy Host que você criou para novochat.duckdns.org.
  3. Na aba SSL:
    • Em SSL Certificate, selecione o certificado que você acabou de gerar (ele deve aparecer na lista com o nome do seu domínio, ex: novochat.duckdns.org).
    • Marque a opção Force SSL (Isso garante que todas as requisições HTTP sejam redirecionadas para HTTPS).
    • Marque HTTP/2 Support.
    • Marque HSTS Enabled.
  4. Clique em Save.

Passo 7: Ajustar Variáveis de Ambiente do Serviço Ticketz

Com o HTTPS configurado, o frontend do Ticketz precisa ser informado para usar o novo domínio HTTPS para se comunicar com o backend. Erros de Mixed Content (conteúdo misto) ocorrem quando uma página HTTPS tenta carregar recursos via HTTP.

  1. No seu servidor, navegue até a pasta do seu serviço Ticketz (ex: /opt/ticketz-docker-local).

  2. Edite o arquivo .env-frontend (ou o arquivo de ambiente correspondente ao seu frontend):

    • Remova ou comente quaisquer variáveis que forcem o protocolo http ou portas específicas (ex: 3011).
    • Atualize as URLs do backend para usar o novo domínio HTTPS. Um exemplo de configuração otimizada seria:

    ini FRONTEND_HOST=novochat.duckdns.org REACT_APP_BACKEND_URL=https://novochat.duckdns.org/backend BACKEND_HOST=novochat.duckdns.org BACKEND_PROTOCOL=https BACKEND_PORT=443 BACKEND_PATH=/backend

  3. Edite o arquivo .env-backend (ou o arquivo de ambiente correspondente ao seu backend):

    • Atualize as variáveis BACKEND_URL e FRONTEND_URL para usar https://novochat.duckdns.org.

  4. Recrie os containers para que as novas variáveis de ambiente sejam aplicadas:

    bash docker compose up -d --force-recreate

Passo 8: Configurar Custom Location para o Backend no NPM (Opcional, mas Recomendado)

Se o seu frontend do Ticketz tenta acessar o backend através de um caminho específico (ex: /backend), você precisará configurar uma Custom Location no Nginx Proxy Manager para rotear essas requisições corretamente.

  1. No painel do NPM, edite o Proxy Host para novochat.duckdns.org.
  2. Vá para a aba Custom Locations.
  3. Clique em Add Location.
  4. Preencha da seguinte forma:
    • Define Location: /backend
    • Scheme: http
    • Forward Hostname / IP: backend (Este é o nome do serviço backend no seu docker-compose.yaml do Ticketz).
    • Forward Port: (A porta interna que o seu backend utiliza, ex: 8080 ou 4000).
  5. Clique em Save.

Passo 9: Limpeza de Cache do Navegador e Teste Final

Mesmo após todas as configurações, o navegador pode estar com informações em cache que impedem o acesso correto.

  1. Verifique a URL: Certifique-se de que você está acessando o site com https:// (ex: https://novochat.duckdns.org).
  2. Force SSL no NPM: Verifique novamente se a opção Force SSL está marcada no seu Proxy Host no NPM.
  3. Limpe o cache do navegador:
    • Abra uma Aba Incógnita (Ctrl+Shift+N) e tente acessar https://novochat.duckdns.org.
    • Se o problema persistir, limpe o cache e os dados do site para o seu domínio nas configurações do navegador.
    • Limpeza de HSTS (Chrome/Zorin OS): Digite chrome://net-internals/#hsts na barra de endereços, vá em Delete domain security policies, digite novochat.duckdns.org e clique em Delete.
  4. Verifique Permissões do Microfone: Clique no ícone de cadeado ao lado da URL no navegador e verifique se o microfone está permitido para o site.

Conclusão

Ao seguir estes passos, você terá configurado com sucesso o Nginx Proxy Manager para atuar como um proxy reverso seguro para o seu serviço Ticketz, utilizando certificados SSL do Let's Encrypt via DNS Challenge do DuckDNS. Isso não apenas garante a segurança da sua aplicação, mas também resolve problemas de funcionalidade em navegadores que exigem HTTPS para acesso a recursos como o microfone.

Lembre-se de que a paciência e a verificação cuidadosa de cada etapa são fundamentais para o sucesso da configuração. Em caso de dúvidas, consulte os logs do Nginx Proxy Manager e do Docker para identificar a causa raiz de qualquer problema.
