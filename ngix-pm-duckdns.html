Guia de Implementação: Novochat (Ticketz)

Infraestrutura com Docker, Nginx Proxy Manager e SSL via DuckDNS

1. Preparação da Infraestrutura

1.1Criando Domínio no DuckDNS

Acesse o painel do DuckDNS e registre o subdomínio novochat. Aponte o IP para o endereço do seu servidor (neste caso, o IP do túnel NetBird 100.116.5.168).

1.2Deploy do Nginx Proxy Manager (NPM)

Configure o NPM em uma rede Docker isolada chamada nginx-proxy para facilitar o roteamento interno.

networks:
  default:
    name: nginx-proxy
    external: true

1.3Configurando o NPM

Acesse o painel administrativo na porta 81. O NPM atuará como a porta de entrada única para o tráfego 80 (HTTP) e 443 (HTTPS).

1.4Configurando o Certificado SSL

Gere um certificado Let's Encrypt usando o DNS Challenge. Utilize o seu Token do DuckDNS para validação automática.

  • Propagation Seconds: Ajustado para 120s para garantir estabilidade.
  • Sintaxe do Token: dns_duckdns_token=seu_token_aqui.

1.5Configurando o Proxy Host

No NPM, adicione um novo Proxy Host para novochat.duckdns.org:

  • Forward IP: frontend (nome do serviço no docker).
  • Forward Port: 80.
  • Websockets Support: Ativado (Essencial para o chat).
  • SSL: Force SSL e HTTP/2 Support ativados.

1.6Configurando Custom Locations (Opcional)

Caso o backend precise de uma rota específica, configure o path /backend apontando para o container backend na porta 4000.

2. Configuração do Serviço Novochat

2.0Ajuste do Ambiente (.env)

Para evitar erros de Mixed Content (bloqueio de scripts inseguros), os arquivos .env-frontend e .env-backend foram atualizados:

REACT_APP_BACKEND_URL=https://novochat.duckdns.org/backend
BACKEND_PROTOCOL=https
FRONTEND_HOST=novochat.duckdns.org

2.1Considerações Finais

"A implementação do HTTPS foi o fator decisivo para a ativação das APIs de mídia (microfone), garantindo que o navegador autorizasse o uso de dispositivos de entrada para mensagens de áudio."

Com esta configuração, o sistema Ticketz opera de forma totalmente segura, centralizada no Nginx Proxy Manager, sem a necessidade de expor portas vulneráveis (como a 3011) diretamente no host.
